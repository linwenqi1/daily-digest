## 2. [The Coming Loop](https://lucumr.pocoo.org/2026/6/23/the-coming-loop/)

来源：**Hacker News** · ⬆ **22** points · 💬 **7** comments

### 🔑 关键概念

- Agent Loop（代理内循环）：编码代理内部运行的循环，模型调用工具（如读文件、编辑、运行测试）、整合结果、继续调用，直到生成回答。这是已熟悉的循环。
- Harness Level Loop（外部控制循环）：在代理循环之外的更高层循环，负责将任务放入队列，由机器尝试执行，然后判断任务是否真正完成；若未完成，则在同一会话中注入新消息、开新会话或交由另一台机器继续，使任务生命周期超越模型自身的“我已完成”终点。
- 防御性代码倾向：当前模型在编写代码时倾向于添加过多本地防御和回退处理，而不是通过强不变量从源头消除错误状态，导致代码复杂度上升、可理解性下降，这在外部循环的持续迭代中被进一步放大。
- 机械翻译与短生命周期产物：循环模式在代码移植、性能实验、安全扫描等不生成全新长期代码、而是转换已有代码或生成短生命期验证产物的场景中表现出色，因为这些工作可通过结果是否等价或是否达成目标来判断，不依赖代码的长期可维护性。
- LLM 作为裁判：在外部循环中，常用另一个 LLM 作为评判者或编排者，其信号不必是客观二值的，只需能驱动下一轮迭代即可，这让代理在实验性工作流等任务中能自主继续。

### 📝 文章摘要

作者观察到越来越多的开发者在编码代理之上构建“外部循环”，让机器自主反复尝试任务，直到外部控制器判定完成；这一模式正成为代理工程中的热门讨论话题。作者个人在编写自己重视的长期代码时使用该模式效果不佳，因为模型生成的代码往往过于防御、复杂、缺乏强不变量，而循环会放大这种问题，且目前未见明显改善。然而，在代码移植、性能探索、安全扫描等不要求代码长期维护或仅做机械转换的场景中，外部循环已展现出令人印象深刻的成果。文章指出，这种转变可能正在将软件从确定性机器推向一种“有机体”隐喻，但作者对是否将完全脱离人类理解的发展方向感到矛盾。

### 💬 社区讨论

有评论认为引用 Claude Code 创造者的话缺乏说服力，并表示对全机器驱动的未来感到不安，甚至因 AI 而停止业余写代码，不再享受所交付的工作。另一些评论者分享了自己在陌生语言或库中用代理写代码的经验，认为代理能帮助解释代码，缓解理解问题。还有评论质疑这种“高级空谈”的商业价值，要求看到对应的十亿美元级公司或利润增长，但被反驳为新技术尚在早期，不应要求立即达到极限速度。

## 5. [Why Drawing Tablet Brands Won't Collaborate on Linux FLOSS Drivers](https://www.davidrevoy.com/article1154/why-drawing-tablet-brands-wont-collaborate-on-linux-floss-drivers)

来源：**Lobsters**

### 🔑 关键概念

- FLOSS 驱动：指自由/开源软件（Free/Libre and Open Source Software）驱动，允许用户自由使用、修改和分发，本文特指在 GNU/Linux 下驱动绘图板的开源方案。
- udev-hid-bpf 项目：由 Red Hat 开发者主导的机制，利用 BPF 在用户空间处理人机接口设备（HID）事件，无需编译内核即可为绘图板添加高性能的开源驱动支持。
- libwacom 与 Wacom 品牌遗产：用于存储多品牌绘图板设备描述的开源仓库仍以“Wacom”命名（如 libwacom、wacom-hid-descriptors），这是 Linux 历史形成的命名惯例，导致其他品牌认为是在为竞争对手贡献数据。
- 设备 HID 描述符：绘图板通过 USB HID 协议上报的按钮、笔压感、分辨率等能力数据，开源驱动依赖这些描述符来正确识别和配置设备，品牌方担心直接共享给“Wacom”命名的基础设施会泄露给竞争对手。

### 📝 文章摘要

作者尝试推动绘图板品牌直接向 Linux 开源驱动项目提交硬件规格，以加速 FLOSS 驱动开发。经多次沟通，负责 Gaomon、Huion、XPPen、Ugee 等品牌技术研发的同一团队最终以礼貌方式拒绝，核心原因是相关开源仓库大量以竞争对手“Wacom”命名，参与即意味着向 Wacom 相关的项目提交自家设备描述符，品牌方无法接受。这暴露了 Linux 绘图板驱动生态中历史遗留的命名问题，已经构成厂商协作的现实障碍。作者因此被迫回归以往逐个评测、抓取描述符的方式，并计划撰写教程引导社区上报规格，当前已有三款新平板等待评测。

## 7. [In memory of the man who put red and green squiggles under words](https://devblogs.microsoft.com/oldnewthing/20260622-00/?p=112451)

来源：**Lobsters**

### 🔑 关键概念

- 托尼·克鲁格（Tony Krueger）：微软 Word 早期核心开发者，从 Word 1.0 起参与多个版本，可能保持着参与发行最多 Word 版本的纪录。
- 阻塞式拼写检查：早期 Word 的拼写检查需要用户手动触发，且会阻塞程序运行，用户必须等待检查完成才能继续操作，常在不当时机打断工作。
- 红色/绿色波浪线：托尼将拼写检查改为后台运行，发现潜在拼写错误后立即在单词下方画出红色波浪线，后来扩展到用绿色波浪线标注语法问题，实现即时、不打断的反馈。
- 无源代码逆向工程：托尼在没有 MS-DOS 版源代码的情况下，通过逆向工程将游戏《Chip’s Challenge》移植到 Windows。
- 后台非侵入式设计：拼写检查不再阻塞前景工作，转为静默扫描，发现问题时立即以波浪线提示，用户无需主动调用即可实时看到纠错建议。

### 📝 文章摘要

微软老兵托尼·克鲁格近期去世，他的工作几乎无人不知但本人鲜为人知。在 Word 早期，他将拼写检查从阻塞操作改造成后台运行，发明了在疑似拼错单词下即时绘制红色波浪线（以及后续的绿色语法波浪线），这一设计后来成为所有文字处理器的标配。他还创造了逆向工程移植《Chip’s Challenge》的记录，并可能保持着参与最多 Word 版本的纪录。他的贡献得到了 Penn & Teller 和“怪人奥尔”等流行文化人物的喜爱与签名致敬，其影响延续至今。
