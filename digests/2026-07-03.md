## 2. [Virginia bans sale of geolocation data](https://www.hunton.com/privacy-and-cybersecurity-law-blog/virginia-bans-sale-of-geolocation-data)

来源：**Hacker News** · ⬆ **841** points · 💬 **130** comments

### 🔑 关键概念

- VCDPA（弗吉尼亚消费者数据保护法）：弗吉尼亚州的一部综合性隐私法律，此次通过S.B. 388修正案新增了禁止销售地理位置数据的条款。
- 地理位置数据销售禁令：该修正案禁止控制者以货币对价向第三方出售能够识别个人位置的数据，但将“销售”定义得比多数其他州的隐私法更窄。
- “销售”的狭义定义：VCDPA将“销售”限定为仅以货币作为对价交换个人数据，不包含以其他有价对价交换的情形，与马里兰、俄勒冈等州将“其他有价对价”纳入定义的做法不同。
- 生效日期：该禁令于2026年7月1日正式生效，在州长2026年4月13日签署后有一小段准备期。
- 精准度限制：根据社区评论，法案实际禁止的是可识别个人在1750英尺（约533米）范围内的精确位置数据，模糊地理位置数据的销售仍被允许。

### 📝 文章摘要

2026年4月13日，弗吉尼亚州长签署了S.B. 388法案，修订《弗吉尼亚消费者数据保护法》，禁止销售地理位置数据。该禁令将于2026年7月1日生效，使弗吉尼亚成为继马里兰和俄勒冈之后第三个禁止此类销售的州。值得注意的是，VCDPA对“销售”采用狭义定义，仅指以货币作为对价的交换，这比其他州的“货币或其他有价对价”定义范围更小，可能为数据流通留下空间。这一立法动向紧随加州总检察长2025年对位置数据行业的调查以及2024年联邦贸易委员会与某数据经纪商的和解行动，反映出监管层对地理位置数据销售的日益关注。

### 💬 社区讨论

社区讨论中存在几个主要分歧点。多位评论者指出法案只禁止精确地理位置数据的销售，模糊位置数据（无法在1750英尺内识别个人）仍可出售，认为这削弱了保护力度。有用户对跨州执法提出质疑：若特拉华州注册的公司出售在弗吉尼亚收集的位置数据，且该公司在弗吉尼亚无运营实体，法案如何执行？回复认为将进入跨州诉讼或联邦法院处理。部分用户讨论了地理位置数据的市场价值，但对具体定价仍缺乏清晰认识。此外，有评论指出加州法律将所有数据使用都泛化为“销售”反而导致规则模糊，而弗吉尼亚的狭义定义或许更明确，但也引发了对规避行为的担忧。用户普遍认同，若给予真正知情的非强制选择，人们会拒绝此类数据的收集与出售。

### 🔑 关键概念

- 本地 AI/LLM：指在用户自有硬件（如个人电脑、工作站）上直接运行的人工智能或大型语言模型，无需依赖云端服务。

- 安全港条款：立法中的一种豁免性表述，可在特定条件下为合法行为提供明确保护，避免动辄陷入许可或违法的灰色地带。

### 📝 文章摘要

一个名为“Right to Local Intelligence”的倡议网站提出，美国某些州的新立法可能要求本地 AI 模型的使用需获得许可，从而威胁到个人在自有设备上自由运行、研究和修改模型的现有实践。网站呼吁民众支持在法律中加入明确的安全港语言，以保障本地 AI 的合法所有权、研究、模型修改与开源发布。因正文缺失，具体是哪些法案、提案或监管行为引发担忧未能得到核实，部分社区讨论也指出原网站未提供实质证据。尽管如此，话题突显了本地智能拥护者与云端 AI 服务商之间日益紧张的对立，以及硬件产业（主要 OEM 厂商）对本地大模型计算的大力押注。

## 5. [crustc: Entirety of rustc, translated to C](https://github.com/FractalFir/crustc)

来源：**Lobsters**

### 🔑 关键概念

- rustc：Rust 语言的官方编译器，本项目中指其全部源代码被转换为 C 语言后得到的可构建版本。
- cilly：作者开发的 Rust 到 C 编译器工具链，能够将任意 Rust 代码编译成适配特定 C 编译器的 C 代码，并支持通过网络与目标平台的 C 编译器通信。
- LLVM：rustc 正常使用的编译器后端基础设施，本项目中生成的 C 编译器仍然需要链接 libLLVM.so 才能运行。
- 见证程序：cilly 在编译时生成的小程序，用于自动检测目标 C 编译器是否支持某种特性（如 _Thread_local），从而让生成的 C 代码与目标平台保持最大兼容。
- ABI 兼容性：指不同编译单元之间函数调用、数据布局等的二进制接口约定，本项目的生成代码与普通 rustc 编译出的代码在大多数平台上调用约定一致，但在 arm64 等平台存在无法从 C 表达的差异。

### 📝 文章摘要

开发者发布了一个名为 crustc 的项目，将完整的 rustc 编译器翻译为 C 语言源码，并可通过 GCC 与 make 直接构建出可工作的 Rust 编译器。该项目是作者历时三年、历经 14 次尝试后实现的 Rust 到 C 工具链 cilly 的演示，其核心创新在于能通过“见证程序”自动适配各类 C 编译器，甚至支持通过 TCP 与目标平台上的 C 编译器通信。构建出的编译器版本为 rustc 1.98.0-nightly，需要链接 LLVM 共享库，作者在 ARM64 Linux 上使用 GCC 13.3.0 编译，无优化时耗时约 1 分 18 秒。这一工作的主要目标是让 Rust 能够运行在仅提供 C 编译器而缺乏 LLVM/GCC 支持的陈旧或冷门硬件上，从而消除因平台支持而转向 C 语言的阻力。
