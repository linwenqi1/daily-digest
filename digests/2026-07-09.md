## 4. [No leap second will be introduced at the end of December 2026](https://datacenter.iers.org/data/latestVersion/bulletinC.txt)

### 🔑 关键概念

- 闰秒：为保持协调世界时（UTC）与地球平太阳时一致，在 UTC 中进行的偶尔一秒调整，可以是正闰秒或负闰秒。
- UTC（协调世界时）：以原子时秒长为基础、通过闰秒与地球自转保持近似同步的全球时间标准，广泛用于民用和科学计时。
- TAI（国际原子时）：由全球原子钟连续累积的高精度时间尺度，不包含闰秒，目前比 UTC 快 37 秒。
- 国际地球自转服务（IERS）：负责监测地球自转参数并发布闰秒引入公告的机构，通过定期公报决定 UTC 是否增减闰秒。

### 📝 文章摘要

国际地球自转服务（IERS）发布公告称，2026 年 12 月底不会引入闰秒。这意味着 UTC 与 TAI 之间的差值将继续保持在 -37 秒（即 UTC 比 TAI 慢 37 秒），全球时间系统无需进行额外的一秒调整。近年来闰秒的触发频率明显降低，主要因为地球自转近期呈现加速趋势，抵消了原本所需的累计正闰秒。该决定对依赖高精度时间同步的计算机网络、卫星导航和金融交易系统有直接影响，软件开发者无需为 2026 年底部署闰秒修正。由于未能提取到公告正文，上述信息主要基于标题及关联讨论中的相关数据推断。

社区讨论整体氛围轻松，但有若干值得关注的观点。部分用户调侃公告的措辞和国际地球自转服务这一机构名称，认为“负责时间测量与分配的权威机构”这一开场白极富仪式感。关于地球自转不可预测的原因，多位用户指出天气、地质活动、含水层抽水、水坝修建和冰川融化等人类与自然因素，导致自转速率发生难以精确预测的变化，回应了“为何无法将地球自转精确计算到小数点后多位”的疑问。另一较有技术深度的分歧集中在闰秒的抽象层级：有用户主张闰秒应像夏令时一样放在时区层处理，但反对者指出闰秒与闰日不同，不同时区并不争议 2 月 28 日后是 3 月 1 日，而闰秒是插入秒的跨区域统一调整，不能简单下移到时区。此外还有幽默调侃，如提议沿赤道安装喷气发动机控制地球自转，或让超人绕地飞行改变时间，以及开玩笑说这样将多活一秒，增加活到 22 世纪的概率。

## 5. [Lobsters Interview with mitchellh](https://alexalejandre.com/programming/interview-with-mitchell-hashimoto/)

来源：**Lobsters**

### 🔑 关键概念

- 终端仿真器：模拟传统硬件终端的软件，负责将命令行程序的输出渲染为窗口内容，并将键盘、鼠标事件传递回程序，是 Ghostty 等项目的核心对象。
- PTY（伪终端）：一种操作系统提供的通信通道，由主、从两端组成，终端仿真器与 shell 进程通过无结构的字节流传递字符和转义序列，其带内信令模式被认为是当前终端应用受限的一大根源。
- Zig：一种注重安全、性能和编译时可计算性的系统编程语言，Mitchell Hashimoto 选择它来实现 Ghostty，目的是在桌面/单机环境中直接操作 GPU、缓存和向量单元，磨练因长期关注分布式侧而钝化的底层技能。
- n-screen API：Mitchell 设想的多屏幕管理协议，允许应用程序创建任意数量的后台屏幕并支持叠放、独立网格尺寸，甚至让屏幕以独立原生窗口呈现，从而突破当前终端仅支持主屏与副屏且切换必须全屏的限制。
- 按钮协议：一种计划中的终端协议，允许应用程序在输出中定义可点击区域并绑定一个消息，即便该区域滚动出可视范围、进入回滚历史，用户点击后仍能将消息送回程序，旨在解决当前鼠标事件仅对屏幕上可见内容有效的缺陷。

### 📝 文章摘要

Hashimoto 在接受 Lobsters 访谈时详细解释了他从 HashiCorp 离开后开发终端仿真器 Ghostty 的初衷：他原计划用 Zig 做一个能运行 Vim 和编译器的个人学习项目，然后弃置，却发现现有终端无法满足快速、功能丰富且原生跨平台的需求，于是将其发展为长期项目并通过私人 Beta 缓慢扩散。访谈的核心在于他对终端生态的批判与改进构思，他认为终端不应被推到什么都做的“极端应用平台”，而应专于文本、可组合和可脚本化的优势，并指出 PTY 的带内信令是核心瓶颈，提出实验性的 n-screen API 和按钮协议，以解决多屏幕管理、历史回滚中交互失效等实际问题。这些观点来自曾主导多个关键云基础设施工具的创作者，可能影响终端和命令行工具链的演进方向。

## 6. [Rewriting Bun in Rust](https://bun.com/blog/bun-in-rust)

来源：**Lobsters**

### 🔑 关键概念

- Bun：一个用 Zig 语言编写的高性能 JavaScript 运行时，集成了打包器、转译器、包管理器和测试运行器，月下载量超过 2200 万次。
- Zig：一种注重手动内存管理和显式控制的系统编程语言，不使用析构函数，而是通过 `defer`/`errdefer` 关键字在作用域结束时显式执行清理代码。
- 垃圾回收(GC)与手动内存混用：Bun 需要同时处理 JavaScript 引擎的垃圾回收对象和 Zig 中手动分配的内存，两者生命周期管理不一致，导致频繁的释放后使用、内存泄漏和双重释放。
- Rust 的所有权与 `Drop`：Rust 通过编译期所有权检查和确定的析构函数(`Drop`)隐式自动释放资源，可以在语言层面系统性防止手动/GC 混合带来的许多稳定性错误。
- AI 辅助重写：重写过程中大量使用 Anthropic 预发布版的 Claude Fable 5 帮助完成从 Zig 到 Rust 的代码迁移。

### 📝 文章摘要

Bun 作者宣布正在将 Bun 从 Zig 全面重写为 Rust。尽管 Zig 让作者一年内一人构建出功能庞大的 Bun，但 Zig 与 JavaScript 垃圾回收混用导致持续出现内存安全问题，仅 v1.3.14 就修复了十余个诸如释放后使用、双重释放等崩溃与泄漏。Bun 月下载量已超 2200 万，被 Claude Code、Vercel 等重要工具和平台集成，团队认为必须从语言层面杜绝此类错误，因此选择迁移到 Rust 利用其所有权和 `Drop` 特性系统提升稳定性。重写过程中使用了预发布版 Claude Fable 5 辅助开发。
