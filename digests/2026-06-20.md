## 1. [CSSQuake](https://cssquake.com/)

来源：**Hacker News** · ⬆ **251** points · 💬 **51** comments

### 🔑 关键概念

- CSS 3D 渲染：利用 CSS 的 3D 变换属性（如 perspective、transform、matrix3d）构造并呈现三维场景，代替传统 WebGL 或 Canvas 渲染管线。
- TypeScript 游戏逻辑：项目使用 TypeScript 实现原版 Quake 的引擎逻辑，包括实体行为、碰撞检测、触发器等，与负责绘图的 CSS 层分离。
- 软件渲染与性能：在缺乏 GPU 加速的纯 CSS 渲染路径下，渲染效率远低于原生引擎，因此出现现代 M1 Pro 机器流畅度不如上世纪 Pentium-133 的现象。
- Quake 引擎行为模拟：实现不仅复刻画面，还尝试重建原版引擎的交互逻辑，但部分机制（如按钮、秘密门触发方式）与原版存在差异。

### 📝 文章摘要

一位开发者创建了名为 CSSQuake 的项目，用 CSS 完成三维场景的渲染，并用 TypeScript 编写游戏逻辑，在浏览器中复原了经典射击游戏《雷神之锤》。社区反馈指出，由于未对性能做专门优化，该版本在 Apple M1 Pro 上的运行流畅度甚至低于上世纪 90 年代的 Pentium-133 电脑。项目展示了 CSS 在三维图形方面的惊人潜力，但同时也让一些玩家更加体会到当年 John Carmack 在早期硬件上所做引擎优化的难度。部分用户还提到类似的 CSSDOOM 项目，并认为这种纯 CSS 的尝试比当下沉重的时事内容轻松许多。

### 💬 社区讨论

社区对项目的评价整体以惊叹和怀旧为主。多数人称赞这一成就是“令人印象深刻的 CSS 使用”，同时指出渲染引擎并非为性能优化，导致流畅度不如 90 年代的低配 PC。有用户发现项目实际依赖 TypeScript 处理游戏逻辑，CSS 只承担渲染，并不完全是“纯 CSS 游戏”。另一些评论聚焦于行为上的细小差异，如部分按钮需要用射击而非触碰来激活，猜测这是完整重建引擎逻辑的结果。还有人调侃“退出比退出 Vim 还难”，并分享了对 CSSDOOM 的喜爱，认为这类项目让人在网络世界重新感受到纯粹的快乐。总结而言，虽然技术上存在性能与非原生行为的问题，社区普遍对这项创意给予肯定，并将其视为对经典游戏与前端技术的一种有趣致敬。

## 4. [Bootimus – A Self-Contained PXE and HTTP Boot Server](https://bootimus.com)

来源：**Hacker News** · ⬆ **50** points · 💬 **18** comments

### 🔑 关键概念

- PXE 网络引导：一种允许计算机通过网络从服务器获取引导镜像并启动操作系统的标准，常用于无盘工作站或自动化系统部署。
- iPXE：开源网络引导固件实现，提供比传统 PXE 更丰富的功能（如 HTTP 引导、脚本化菜单），被 Bootimus 嵌入为默认引导加载程序。
- 代理 DHCP（proxyDHCP）：一种网络配置方式，不改变路由器主 DHCP 服务器的设置，仅额外提供 PXE 所需引导信息，实现无缝切入现有局域网。
- 自包含单二进制：指程序编译为单个静态可执行文件，内部嵌入 Web 界面、SQLite 数据库、iPXE 文件等，无需外部运行时依赖，通过 scp 复制即可运行。
- 自动内核/initrd 提取：Bootimus 能直接从 ISO 中自动提取内核和初始化内存盘文件，支持 Ubuntu、Debian、Arch、Fedora 等 50 余种发行版，简化了手动配置过程。

### 📝 文章摘要

Bootimus 发布了一个自包含的 PXE 和 HTTP 引导服务器，它作为单个 Go 二进制文件分发，内嵌 iPXE、Web UI、SQLite 等所有资源，无运行时依赖。该工具可一键部署（如通过 Docker），在 30 秒内启动一个完整的网络引导环境，并利用代理 DHCP 在不触碰现有路由器配置的情况下响应客户端请求。它能自动识别 50 多种发行版和工具镜像（如 GParted、Clonezilla），支持安全启动、自定义引导加载程序以及无人值守安装文件的注入。开发者强调其 zero-config 和可移植特性，旨在降低批量部署和维修场景中的操作门槛，但原帖未提供具体性能对比或大规模测试数据。

### 💬 社区讨论

多数评论者认可项目的实用性，但围绕其创新性、表现形式和文本风格存在分歧。有用户指出，若已自行管理 DHCP 和 Web 服务器，添加 TFTP 服务实现类似功能并不复杂，因此感到该项目有“重复造轮子”之嫌；同时有人提到 netboot.xyz 和 FOG Project 等成熟替代方案。另一部分人则欣赏其简化配置的尝试，认为在基本使用场景中进一步降低了复杂度，并分享了用 AI 辅助开发类似工具的经历。部分评论聚焦于项目网站的 UI 设计，批评其“表演式 UI”中的绿色状态点等元素显得过时且不必要。此外，多名用户指出项目描述文本带有明显的 AI 生成特征，但均表示这不减损项目本身的价值和兴趣。

## 6. [I can haz smoller NixOS ISOs?](https://natkr.com/2026-06-19-nixos-but-smol/)

来源：**Lobsters**

### 🔑 关键概念

- NixOS 配置构建：NixOS 允许用一组声明式配置来生成完整的系统环境，包括虚拟机（VM）和可启动 ISO 镜像，构建过程通过 nix-build 命令实现。
- 实例 ISO 镜像：文中生成的 ISO 是 NixOS 的一种可独立运行、不依赖宿主机 Nix 存储的映像，包含完整 /nix/store，适合用于 libvirt 或远程部署。
- squashfs 文件系统：压缩只读文件系统，用于存储 ISO 的主用户空间，文章通过挂载 squashfs 镜像来分析其内容大小。
- nix why-depends：Nix 工具，用于追踪软件包之间的依赖关系，作者用它找出为什么某些体积较大的包（如 Boost）被引入 ISO。

### 📝 文章摘要

## 7. [There Are No Instances in atproto](https://overreacted.io/there-are-no-instances-in-atproto/)

来源：**Lobsters**

### 🔑 关键概念

- 实例（Instances）：Mastodon 等联邦社交网络中，将托管与应用捆绑在一起的独立社区节点，用户身份和内容绑定在特定实例，实例之间通过消息转发实现互通。
- 托管与聚合分离：指内容的实际存储位置（个人数据服务器或博客）与抓取、展示内容的终端应用（如 RSS 阅读器）是解耦的，atproto 将此模型用于社交网络。
- atproto（AT Protocol）：一个将内容托管和内容聚合在网络层面切分开的协议，没有实例概念，应用只是整个“大气层”的投影，类似于 RSS 和 Google Reader 的关系。
- 联邦（Federation）：Mastodon 实例间通过相互协定转发帖子的机制，用户跨实例关注会导致 O(n²) 的箭头规模，且可能出现实例撕裂。
- 用户代理：在 atproto 中，用户可以通过更换托管、自建数据存储、或开发与使用不同应用来实现去中心化，不必绑定于某一特定服务端节点。

### 📝 文章摘要

作者撰文澄清 atproto 中不存在“实例”概念，指出这是一个源自 Mastodon 思维模式的类别错误。文章通过将对 RSS 与 Google Reader 的类比引入 social network，说明 atproto 把内容托管和聚合在网络层面彻底分离，每个应用只是整体“大气层”的投影，而不是独立的封闭小社交网络。与此不同，Mastodon 的每一个实例都是托管与应用的捆绑体，用户身份绑定于实例，实例间依赖联邦转发。这种区别意味着在 atproto 中，用户可以通过更换托管、自运行数据服务或创建新应用来实践去中心化，远比复制多个 Bluesky 数据库节点更有意义。

## 11. [Microsoft discovers new lightweight backdoor that steals cryptocurrency](https://arstechnica.com/security/2026/06/microsoft-spots-new-self-propagating-malware-for-stealing-cryptocurrency/)

来源：**Ars Technica**

### 🔑 关键概念

- Crypto Clipper：微软发现的一种自传播蠕虫，通过 USB 驱动器扩散，专门窃取剪贴板中的加密货币钱包地址或助记词，并能远程执行代码，兼具信息窃取与后门功能。
- 剪贴板劫持：恶意软件实时监控设备剪贴板内容，当检测到符合加密货币地址或助记词格式的文本时，自动截取并在 10 秒内连续抓取 5 张屏幕截图。
- Tor 匿名通信：窃取的数据通过 Tor 网络发送到攻击者控制的服务器，利用冗余节点转发流量使日志无法同时记录发送和接收 IP，从而隐藏通信踪迹。
- SOCKS5 代理：Crypto Clipper 使用本地 SOCKS5 代理将自身流量转发到 Tor 客户端，不依赖传统的命令与控制服务器 IP，使检测和阻断更加困难。
