## 1. [Deno Desktop](https://docs.deno.com/runtime/desktop/)

来源：**Hacker News** · ⬆ **589** points · 💬 **216** comments

### 🔑 关键概念

- Deno Desktop：Deno 运行时内置的桌面应用打包工具，能将 TypeScript 文件或 Next.js 等 Web 项目编译为自包含的二进制可执行文件，集成 Deno 运行时与 Web 渲染引擎。
- WebView 后端与 CEF 后端：Deno Desktop 提供的两种渲染后端。默认 WebView 使用操作系统内置组件，生成体积较小的包；可选 CEF（Chromium Embedded Framework）后端可保证跨平台一致的渲染效果，但会使二进制体积增大。
- 框架自动检测：该功能可自动识别 Next.js、Astro、SvelteKit 等主流 Web 框架的生产或开发服务器，无需修改代码即可将现有 Web 项目转为桌面应用，并支持热模块替换。
- 二进制差分自动更新：发布时只需提供一份 manifest 和 bsdiff 补丁文件，运行时自动轮询、应用补丁，并在启动失败时自动回滚，简化桌面应用的更新流程。
- 进程内绑定：后端与 Web UI 的通信通过进程内通道完成，而非跨进程的 socket IPC，减小通信开销，避免序列化与跨进程往返延迟。

### 📝 文章摘要

Deno v2.9.0 引入了非稳定特性 Deno Desktop，可将 Deno 项目打包为各平台自包含的桌面应用。该工具默认使用系统 WebView 以减小体积，同时支持 CEF 后端以保证渲染一致性，并内置框架自动检测、进程内绑定和二进制差分更新机制。社区反馈中，用户测试生成的二进制文件达 442 MB，引发对体积的质疑，官方路线图则提出通过共享 CEF 运行时降低多应用体积。该功能使 Deno 直接竞争 Electron 和 Tauri 等桌面开发方案，可能吸引更多 Web 开发者进入桌面应用领域。

### 💬 社区讨论

主要分歧集中在二进制体积和权限安全上。有开发者测试 Hello World 编译出 442 MB 的包，远大于 Electron 的预期，其他用户指出这是因为使用了 CEF 而非系统 WebView，官方也提到后续将通过共享 CEF 运行时将体积降至数 MB。关于权限系统，有人担心编译时固化权限会让用户误以为二进制受 Deno 沙箱保护，但评论指出开发者分发的二进制本就不保证权限完整性，这种呈现可能具有误导性。此外，多位用户对 Deno 生态的成熟度和这一新功能表示欢迎，认为它

## 2. [Codex logging bug may write TBs to local SSDs](https://github.com/openai/codex/issues/28224)

来源：**Hacker News** · ⬆ **183** points · 💬 **92** comments

### 🔑 关键概念

- SQLite WAL（Write-Ahead Log）：SQLite 的预写日志机制，可提升并发性能，但如果有大量写入操作会显著放大实际写入磁盘的数据量。
- SSD 写入耐久度（TBW, Terabytes Written）：固态硬盘在保修期内可承受的总写入量，消费级常见值为 600 TBW，持续超量写入会快速消耗寿命。
- TRACE 日志级别：最详细的日志级别，会记录所有内部事件（包括 inotify 文件事件、网络帧、协议数据等），默认开启会导致日志体积极度膨胀。
- 写入放大（Write Amplification）：指实际写入存储设备的数据量远大于应用层保留的有效数据，本例中日志被不断插入再清理，导致保留数据仅约 1 GB，但日写入量可达数十 GB。

### 📝 文章摘要

OpenAI 的 Codex 工具因默认以 TRACE 级别记录所有日志到本地 SQLite 数据库，导致极高的 SSD 写入量。一名用户报告，机器持续运行 21 天后，主 SSD 写入量已约 37 TB，据此推算年写入量可达约 640 TB。在 1 TB 消费级 SSD（典型耐久度约 600 TBW）上，这种写入强度可能在不到一年内消耗掉整个驱动器的额定写入寿命。日志中超过 70% 为 TRACE 级别记录，主要来源包括 inotify 事件、WebSocket 原始负载和遥测镜像事件，通过过滤这些类别即可去除约 96% 的日志字节。根本原因是在安装 SQLite 反馈日志接收器时使用了全局 TRACE 默认值，开发者建议将默认日志阈值提升并限制数据库总写入量。

### 💬 社区讨论

- 批评软件质量：多位用户批评 Codex 是“slopware”，指出即使仅显示加载动画也会导致 GPU 占用 100%、界面卡顿、频繁失败等问题。
- 临时修复方法：有人分享了通过 SQLite 触发器阻止日志插入以及使用 VACUUM 命令将数据库从 27 GB 压缩至 73 MB 的应急方案。
- 竞品体验对比：存在分歧，一部分用户认为 Codex 输入延迟高、体验差，而 Claude Code 延迟较低；另一部分用户体验相反。
- 调侃与反思：社区调侃“vibe coding”带来的“快速破坏”，还有人指出将 TRACE 日志默认开启是经典错误，但讽刺的是其破坏方式从拖慢应用变成了直接损耗硬件。

## 3. [Help I accidentally a wigglegram](https://lmao.center/blog/wiggle-accidents/)

来源：**Hacker News** · ⬆ **329** points · 💬 **71** comments

### 🔑 关键概念

- wigglegram：一种立体图像形式，通过循环播放从不同角度拍摄的连续帧（如GIF）来模拟三维深度效果，常被称为“扭动图”。
- 感知哈希（perceptual hashing）：一种图像指纹算法，生成的哈希值会随图像视觉相似度而变化，相似图像具有更接近的哈希位，常用于反向图像搜索。
- 汉明距离（hamming distance）：衡量两个等长哈希值之间不同位数的度量，用于判断图像间的相似程度，距离越小图像越可能属于同一场景的变体。
- 立体视觉（stereoscopy）：利用双眼视差原理，通过向左右眼呈现略有差异的图像来营造深度感知，wigglegram是其中一种无需眼镜的实现方式。
- iCloud照片库：苹果云存储中的照片集合，本文脚本支持直接从Mac上的iCloud照片库读取并处理图像，是意外wigglegram的主要来源。

### 📝 文章摘要

作者发现，因自己拍照时习惯对同一场景拍摄多个不同角度的帧，手机中积攒了大量可制作成wigglegram的素材。为了自动找出这些相似照片序列，他编写了一个脚本，利用感知哈希计算所有图片指纹，再通过汉明距离设置阈值提取匹配对。最终从多年累积的照片中涌出数百个意外生成的wigglegram，其中许多呈现“动态影像”而非纯粹立体感，动物、设计稿和雕塑类场景尤其适合。该脚本已开源在GitHub上，可直接处理Mac的iCloud照片库或指定本地目录，展示了自动化发掘隐藏立体内容的可能性。

### 💬 社区讨论

多位读者分享了延伸项目和观点：有人推荐使用四枚等距树莓派摄像头自制立体相机（k4mera.world），以精确控制视角；有人提到自己编写的mpl_stereo库可用于在matplotlib中制作wigglegram和立体图，并认为伪3D是数据可视化中被低估的手段。关于效果，部分用户认为Reddit上固定水平移动且同时拍摄的wigglegram深度感最强，而文中那些带有时间差的照片则显得像脱节的动画，额外运动反而削弱了立体感。也有评论称赞脚本“手工感”十足，在AI生成代码泛滥的当下令人耳目一新。另有人指出Nintendo 3DS的双摄像头可制作3D照片，并提供在线转换工具，讨论偏移量对焦点的影响。还有开发者建议将该功能作为Digikam（已支持感知哈希）的插件，并询问是否自行实现了哈希算法。此外，大家对“frontend团队命名”引发的小调侃也做了幽默回应。

## 5. [Chesterton's middle finger](https://www.arp242.net/chestersons-finger.html)

来源：**Lobsters**

### 🔑 关键概念

- Chesterton's fence analogy：切斯特顿栅栏比喻，指在没有理解事物为何如此之前不应贸然改变，因为可能存在尚未考虑的合理理由。
- Chesterton's middle finger：作者引申的概念，指开发者故意不记录做奇怪事情的原因，用无意义的提交信息给后续维护者留下巨大障碍，形同竖中指。
- 提交信息（commit message）：版本控制系统中记录每次代码变更的说明，好的提交信息应该回答“改了什么、为什么改、为什么这样改是好的”三个问题。
- 技术债务与文档缺失：指代码中存在未完成的改造、遗留功能、从未被链接使用的特性等，且没有文档或注释说明，导致接手者难以理解系统。

### 📝 文章摘要

作者通过分析一个持续13年的项目仓库，发现其提交正文总计仅295行，大部分为无意义的“fix page A”之类，有效信息约每月一行。项目没有其他文档或代码注释，留下大量未完成的改造、遗弃功能和无人使用的特性，作者称之为“切斯特顿的中指”——开发者明知做了奇怪的事却故意不解释原因。作者强调，写提交信息不是可选项，而是软件开发的本职工作，哪怕只是粗略记录，也比完全不写对后来者友好得多。

### 🔑 关键概念

- Vibe coding：非专业开发者借助AI对话式工具，口头描述需求直接生成应用，全程无需手写代码，但生成过程缺少安全审查，导致应用易含漏洞。
- SQL注入：一种网站漏洞，攻击者通过在输入字段插入恶意SQL语句，可读取、篡改或删除后台数据库中的敏感数据，常因代码未过滤用户输入而引起。
- 认证缺失：应用未设置登录或身份校验机制，任何人可直接访问本该受保护的功能或数据，是vibe-coded应用中常见且危害大的安全问题。
- 威胁模型：针对具体应用梳理“谁会攻击、怎样攻击、会造成什么后果”的分析方法，vibe coding时往往被跳过，导致应用在云端暴露时风险失控。
- 代码代理安全审查的局限性：Claude Code、Codex等AI编码工具虽内置安全扫描命令，但大多需要用户主动触发，非专业用户容易忽视，形成虚假的安全感。

### 📝 文章摘要

本文报道了“vibe coding”热潮下，非开发者用AI快速生成应用带来的安全隐患。一个名为“Boomberg”的个人网站上线数月后才发现隐藏的SQL注入漏洞；其他案例中，AI编码代理删除了生产数据库，黑客轻易入侵未加防护的演示应用，AI社交平台Moltbook更是将包含数万条邮箱和私密消息的生产数据库直接暴露在公网。安全公司Red Access发现约5000个由vibe coding工具构建的公开应用中，近2000个泄露了医疗、财务等敏感数据。专家指出，个人应用一旦涉及他人数据或被放上云端，就必须提高安全标准，但多数用户不了解威胁模型，也不会主动触发AI工具的安全审查，导致“容易构建但难以保安”的风险急增。
