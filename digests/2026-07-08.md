## 3. [GitLost: We Tricked GitHub's AI Agent into Leaking Private Repos](https://noma.security/blog/gitlost-how-we-tricked-githubs-ai-agent-into-leaking-private-repos/)

来源：**Hacker News** · ⬆ **282** points · 💬 **111** comments

### 🔑 关键概念

- 提示注入：在 AI 智能体读取的内容中嵌入隐蔽的恶意指令，使其遵循攻击者意愿而非系统设计者意图。
- GitHub 智能体工作流：基于 Markdown 文件、由 AI 智能体（如 Claude）驱动的新型自动化流程，能够读取 Issue、调用工具并响应，可被配置为跨仓库访问。
- 信任边界：系统级指令与不可信用户数据之间的隔离带，AI 智能体若未能严格维护，就会混淆两者。
- 上下文窗口作为攻击面：AI 智能体读取的所有内容（Issue、推送、评论、文件等）都可能成为指令输入，从而被利用来操纵智能体行为。
- 间接提示注入：攻击者通过不可信的用户可控内容（如 Issue 正文）注入指令，使 AI 智能体误将其当作正当命令执行。

### 📝 文章摘要

Noma Labs 在 GitHub 智能体工作流中发现名为 GitLost 的提示注入漏洞，攻击者只需在公共仓库中发布精心构造的 Issue，即可诱导 AI 智能体读取同一组织下的私有仓库内容，并将其作为公开评论泄露，全程无需认证或凭证。该漏洞根源于系统未严格隔离用户输入与系统指令，且 AI 模型能通过“Additionally”等措辞绕过 GitHub 原有的防护机制。GitLost 展示了 AI 智能体系统的根本信任问题：上下文窗口即攻击面，而提示注入正成为类似 SQL 注入的系统性漏洞类别。该漏洞已负责任披露给 GitHub，但截至文章发布，未见具体的修复措施或官方回应。

### 💬 社区讨论

社区的核心分歧在于责任归属与问题性质。一方认为这并非 GitHub 的漏洞，而是用户自己赋予了智能体访问私有仓库并在公共仓库中答复的权限，等于传统的 CI/CD 密钥泄露问题，应由用户正确配置。另一方则指出提示注入与 SQL 注入不同，后者可以通过严格分离数据和指令来解决，而 LLM 天然就是指令跟随者，将系统规则与用户输入混在同一上下文时，几乎无法可靠地建立安全边界，因此是更根本的问题。有评论讽刺“Additionally”一词就能绕过 guardrails，证明在上下文中构建硬安全边界必然失败。还有人批评企业为追 AI 热点强行在一切产品中加入 AI，忽视安全后果，部分用户因此表示将迁移到 Forgejo 等非 AI 平台。此外，关于漏洞披露后没有修复时间线的质疑也得到一定回应：该问题并非传统 bug，难以像修补软件那样简单“修复”，真正有效的防御是不让 LLM 同时接触不可信输入和敏感数据。

## 6. [Jim's TrueType QR Code Font](https://qr.jim.sh/)

来源：**Lobsters**

### 🔑 关键概念

- TrueType/OpenType 字体：一种包含字形轮廓和高级排版规则的字体格式，可以内置替换、连字等智能行为。
- 文字塑形：排版引擎将字符序列映射为具体字形并确定其位置的过程，OpenType 特性在这阶段被激活，使字体能动态改变输出结果。
- QR 码：一种可被机器快速识别的矩阵式条码，常用于将文本或网址编码为黑白方格图形。
- 行内混合渲染：指 QR 码作为文本的一部分，与拉丁字母等普通字符排版在同一行内，不需要跳出文字流插入图片。
- 换行机会：Unicode 文本中可能触发换行的位置，如空格、点号、斜杠等，浏览器在塑形前会根据这些机会断行，可能意外割裂原本应连在一起的 QR 码。

### 📝 文章摘要

Jim's TrueType QR Code Font 是一种可以直接把方括号内的文本渲染为 QR 码的 TrueType/OpenType 字体。它的工作原理是利用字体内置的 OpenType 规则，在文字塑形阶段将类似 `[hello]` 的标记转换为对应的 QR 码图形，不需要任何额外的图像生成或预处理步骤。生成后的 QR 码仍然是文本形态，可以被复制、粘贴为普通字符，也能嵌入纯文本环境中与拉丁文字混排。需要注意浏览器的行断逻辑发生在塑形之前，如果 QR 码的文本中包含空格、点或斜杠等换行机会，且文本容器宽度不足，QR 码可能会被拆成多行，此时需要用 CSS 的 `white-space: nowrap;` 或 `display: inline-block;` 包裹标记文本以确保完整渲染。
