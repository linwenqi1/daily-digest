## 1. [A website that lists websites to submit your website to](https://www.submission.directory/)

来源：**Hacker News** · ⬆ **121** points · 💬 **27** comments

### 🔑 关键概念

- 网站提交目录：一种聚合了大量可提交网站链接的列表页，用户通过将自身网站提交到这些平台上，以换取曝光、流量或外链。
- nofollow 与 dofollow 链接：nofollow 是告知搜索引擎不要传递权重的链接属性，对排名直接影响小；dofollow 则允许权重传递，是 SEO 中外链建设的主要目标。
- 反向链接：其他网站指向你的网站的链接，被搜索引擎视为权威性投票，获取高质量反向链接有助于提升搜索排名。
- AI 生成答案的可见度：随着搜索引擎整合 AI 直接给出答案，网站被 AI 模型作为信息来源引用，逐渐成为新的流量渠道，重要性不亚于传统排名。
- 手选策划目录：由人工审核挑选收录网站的目录，区别于自动聚合，因其筛选标准通常被认为更具可信度和信号价值。

### 📝 文章摘要

一个名为 submission.directory 的网站整理了一份“网站提交网站”的清单，收录了 Product Hunt、Hacker News、Capterra 等数十个可提交产品、业务或初创公司链接的平台，并逐一标注了链接属性（nofollow/dofollow）、收费模式和受众特点。这份清单的价值在于将分散的推广渠道集中呈现，帮助创业者一站式发现可获取反向链接、早期用户和媒体关注的地方，尤其强调了部分平台在搜索引擎和 AI 回答中的高曝光潜力。不过，页面内未提供具体数据或效果统计，仅为渠道罗列和简要点评，需谨慎看待实际转化效果。

### 💬 社区讨论

评论者普遍意识到这类汇总清单会加剧垃圾提交，有用户直言“公开发布这种清单是垃圾邮件的配方”，并建议应优先将网站分享给真实社区而非机械提交。也有怀旧声音，将之与 90 年代的 Submit It 或 StumbleUpon 类比，指出现在挑战已从搜索引擎收录转向触达特定受众。部分讨论围绕 SEO 诚实性展开，认可目录承认其链接价值，同时指出随意提交仍可能被视为不礼貌。此外，有人提出一个有趣的观点：随着 AI 取代搜索引擎，人工策展的主题目录可能复苏，甚至可能出现经认证不含 AI 内容的网站列表。还有用户提供了包含 1057 个站点的更大列表，以及经典的递归玩笑。

## 2. [I found 10k GitHub repositories distributing Trojan malware](https://orchidfiles.com/github-repositories-distributing-malware/)

来源：**Hacker News** · ⬆ **293** points · 💬 **84** comments

### 🔑 关键概念

- 木马分发仓库：攻击者在 GitHub 上创建的大量仓库，通过伪造热门项目的提交历史和描述，在 README 文件中加入指向恶意 zip 压缩包的链接，以分发木马软件。
- 模式匹配检测：作者通过观察恶意仓库共同行为（如每隔几小时删除上一个提交、推送仅修改 README 的新提交，且其中包含指向压缩包的链接）来定义搜索模式，并用脚本自动筛选匹配仓库。
- gharchive：一个提供 GitHub 每日事件归档的服务，此研究用它来快速筛选出近期有推送事件的仓库，而不必遍历全部 5 亿个仓库，从而将分析范围从 1600 万次推送缩小到 4 万个候选。
- 二进制与源码不一致：攻击者利用开源平台的信任，复刻合法项目的提交历史（包括原作者为贡献者），但在 README 中添加恶意压缩包，压缩包中的可执行文件在 VirusTotal 直接上传时会被检测到木马，而仅扫描链接则无风险。
- 规避检测的频率更新：恶意仓库通过每隔几小时删除旧提交并重新推送相同内容，使其在“最近更新”排序中靠前，增加被搜索或代理自动拉取的概率，这一行为特征成为识别它们的关键切入点。

### 📝 文章摘要

一名安全研究人员发现 GitHub 上存在大规模木马分发活动。他通过自己仓库被恶意复制并添加恶意链接的经历，总结出恶意仓库的行为模式：每隔数小时删除旧提交，推送仅修改 README 且包含 zip 压缩包链接的新提交，且这些仓库均非 fork，拥有不同贡献者和名称。借助 gharchive 与 GitHub API，他编写脚本从 40000 个频繁更新的仓库中识别出约 10000 个符合完整模式的仓库，这些仓库均在 README 中指向包含木马的压缩包。这些恶意仓库已存在数月甚至超过一年，GitHub 未能自动检测和删除。该发现揭示了攻击者利用平台信任和搜索排名机制，通过伪装合法项目大规模分发木马的威胁，对依赖开源代码的开发者构成严重风险。

### 💬 社区讨论

社区讨论聚焦于攻击意图与防御启示。有评论指出，这种模式可能主要针对自动搜索并拉取依赖的 AI 代理，而非人类开发者，因此恶意仓库只需在少数搜索中命中即可得手。部分用户将此案例联系到 Bing 搜索中假冒仓库的排名问题，有人提到使用 uBlock Origin 等过滤工具防御。关于“开源等于安全”的假设被批评为不现实，因为审查代码和验证二进制匹配需要大量精力，而 GitHub 已成为软件分发中心，潜在风险增大。另有一位用户上传样本到分析平台，发现该木马与 Disco 木马家族相关，提供了技术关联。还有评论赞同频繁更新是为了在“最近更新”排序中置顶，从而诱使用户克隆恶意仓库。此外，有人借 Disney 工程师中招的案例，提醒不应将多因素认证令牌存放在密码管理器中。这些讨论从攻防角度强调了提高供应链安全意识的重要性。

## 7. [Changes in Emacs 31 I'm Already Daily Driving](https://rahuljuliato.com/posts/emacs-31-around-the-corner)

来源：**Lobsters**

### 🔑 关键概念

- tree-sitter：一种增量解析库，Emacs 31 将其深度集成，可为编程语言提供更精确的语法高亮和代码结构分析，并支持自动获取和编译语法文件。
- treesit-auto-install-grammar：Emacs 31 新增的变量，设为 t 后，当缺少某语言的 tree-sitter 语法文件时，Emacs 会自动提示下载并编译，省去原先手动配置的步骤。
- markdown-ts-mode：Emacs 31 内置的实验性 Markdown 编辑模式，基于 tree-sitter，提供接近 Org 模式的键绑定、代码块实况高亮、内嵌图片显示等功能，尚未自动关联 .md 文件。

- 跨架构语法二进制不兼容：自动安装的 tree-sitter 语法库以相同文件名存放在不同架构机器上，x86_64 与 arm64 的 .so 文件会相互覆盖，导致在另一架构上无法加载，是一个需要注意的潜在陷阱。

### 📝 文章摘要

作者正在日常使用从开发分支构建的 Emacs 31，并介绍了其中两个已融入其配置的关键改进。最重要的变化是 tree-sitter 体验大幅简化：设置 `treesit-auto-install-grammar` 和 `treesit-enabled-modes` 后，Emacs 可自动获取和编译语法文件并切换至对应模式，大量原先的手动配置将被删除。另一个亮点是内置的实验性 `markdown-ts-mode`，由作者发起并与 Stéphane Marks 合作开发，提供类似 Org 的编辑体验、代码块实况语法高亮和行内图片显示，但需手动启用，且代码块内补全仍有缺陷。请注意此内容基于 2026 年中开发状态，最终发布时细节可能变动。
