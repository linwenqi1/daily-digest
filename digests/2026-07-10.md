## 3. [Show HN: Getting GLM 5.2 running on my slow computer](https://github.com/JustVugg/colibri)

来源：**Hacker News** · ⬆ **742** points · 💬 **180** comments

### 🔑 关键概念

- 混合专家模型（MoE）：GLM-5.2 是一个 744B 参数的大型语言模型，Token 生成时只激活其中约 40B 参数，其余大量参数组成 21,504 个路由专家，按需从磁盘动态加载。
- int4 量化：将模型参数压缩为每个权重 4 位整数存储，大幅降低磁盘占用（约 370 GB）和常驻内存（约 9.9 GB），在解码时按需反量化，平衡精度与资源消耗。
- 流式专家加载与缓存：将路由专家保存在磁盘上，通过分层 LRU 缓存和可选的热专家固定区（pinned hot-store），按 Token 需求从 SSD 流式读取所需专家，并利用操作系统的页缓存加速重复访问。
- MTP 推测解码：利用模型自身的多 Token 预测头（MTP）一次性草拟多个候选 Token，再通过主模型批量验证，在命中率较高时可将有效生成效率提升至每个前向步骤 2.2–2.8 个 Token，且通过拒绝采样保持输出无损。
- C 语言无依赖引擎：整个推理引擎由纯 C 语言实现（约 2400 行），无 BLAS、无 Python 运行时、无 GPU 强制依赖，可在消费级 CPU 上运行，通过 AVX2 整数点积内核加速矩阵运算。

### 📝 文章摘要

开发者在仅有约 25 GB RAM 的消费级计算机上，成功运行了具有 744B 参数的 GLM-5.2 混合专家模型。其核心方法是：将模型以 int4 量化格式存储于约 370 GB 的磁盘上，仅把密集部分常驻于约 9.9 GB 内存，其余路由专家按需从 SSD 流式加载，并辅以专家缓存与 MTP 推测解码来缓解磁盘读取延迟。该方案使得前沿大模型能在低成本硬件上推理，尽管冷启动时速度极慢（约 0.05–0.1 tok/s），但通过预热缓存和推测解码，可将单次前向 Token 产出提升至 2.2–2.8 个，验证了“去 GPU”的可行性，但并未声称实现实用交互速度。关键数据包括：内存峰值自动控制在约 20 GB，模型磁盘占用约 370 GB，冷 Token 解码约需 11 GB 磁盘随机读取。

### 🔑 关键概念

- 双螺旋扭转柱：教堂内部的四种柱子均由两根反向的所罗门柱交织而成，截面为星状多边形，柱头分叉如树枝，支撑起塔楼与屋顶。

### 📝 文章摘要
