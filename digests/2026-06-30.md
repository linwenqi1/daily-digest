## 4. [Parse, Don't Validate – In a Language That Doesn't Want You To](https://cekrem.github.io/posts/parse-dont-validate-typescript/)

来源：**Hacker News** · ⬆ **33** points · 💬 **11** comments

### 🔑 关键概念

- 解析与验证：验证器只是判断数据是否有效并返回布尔值，但类型系统不会记住这次判断，之后仍需重新检查；解析器则尝试将原始数据转换成一个更精确的新类型（如从 `string` 生成 `Email`），并将“已校验”这一信息编码在类型中，后续代码无需再次怀疑。
- 品牌类型（Branded Types）：在 TypeScript 这种结构类型系统中，通过交叉类型 `string & { readonly [BrandSymbol]: true }` 创建一个编译时独有的“幻影”标记。这个标记只在类型层面存在，运行时没有实际字段，外部无法伪造，从而模拟出名义类型的效果，迫使数据必须通过解析函数才能获得。
- 结构类型与名义类型的差异：TypeScript 使用结构类型，只要两个类型形状相同就视为同一类型，这使得无法直接区分 `string` 和 `Email`。Haskell 等语言支持名义类型（如 `newtype`），可以原生地创建真正不同的类型。品牌类型是对这一缺失的补丁。
- 解析器返回的联合类型：解析函数不是抛异常或返回布尔值，而是返回一个带标签的联合类型，如 `{ kind: "ok"; value: T } | { kind: "err"; error: ParseError }`。这让成功与失败路径对类型系统完全可见，迫使调用方显式处理两种结果，避免不可见的控制流。

### 📝 文章摘要

本文在 TypeScript 的语境下深入阐释了“解析，不要验证”这一原则。作者指出，常规的验证函数（返回 `boolean`）在检查完成后便丢弃了信息，导致程序中到处散布重复检查；而解析器则通过把校验结论固化到更精确的类型（如 `Email` 类型）中，让后续代码天然安全。文章重点展示了如何利用品牌类型（`string & { readonly [symbol]: true }`）在 TypeScript 中模拟名义类型，从而强制数据必须经由解析函数构造，并在解析函数中使用带标签的联合类型来显式传递错误，而非依赖隐式的异常。这种方法使“非法状态不可表示”在 TypeScript 中成为可能，尽管语言本身并不主动鼓励这种模式。

### 💬 社区讨论

社区中存在多角度的讨论。一部分读者怀疑文章内容或文风有较明显的 AI 生成痕迹，并引发了是否需要披露 AI 参与度的争论；也有读者认为即便有此嫌疑，文章对品牌类型和“解析优于验证”的解释仍清晰有用，帮助自己简化了代码并提升了可靠性。实践层面的讨论集中于两点：一是对于存在大量可选字段的场景，创建过多细分类型（如 `UserWithPhoneNumber` 等）可能导致类型爆炸，有回复推荐了 Rich Hickey 的演讲“Maybe Not”来思考可选性设计；二是提醒在构造这类类型时，应对解析后的数据对象进行 `Object.freeze`，以避免运行时被意外篡改。此外，有人提到这本质上是领域驱动设计（DDD）的支柱之一，但在团队中推行并不容易。

## 6. [What is `std::pin::Pin` in Rust?](https://vrong.me/blog/what-is-pinning-in-rust/)

来源：**Lobsters**

### 🔑 关键概念

- Pin (std::pin::Pin)：Rust 中的一种指针包装类型，它对所指对象做出“不会通过该指针被移动”的类型级保证，主要用于保护自引用结构中的内部指针不被悬空。
- 自引用类型 (self-referential types)：包含指向自身字段的指针或引用的结构体，一旦实例在内存中被移动，内部指针便会指向旧地址而失效，是 Pin 引入的核心动机。
- Unpin：Rust 中的自动标记 trait，表示类型在被固定后仍可安全地恢复普通可变引用（即不依赖固定保证）。绝大多数标准类型都自动实现 Unpin；若类型中包含自引用且未显式排除 Unpin，则安全恢复 `&mut T` 可能导致移动并引发未定义行为。
- PhantomPinned：标准库中的标记结构，明确不实现 Unpin，常作为字段嵌入到自定义自引用结构体中，利用自动 trait 传播机制使整个结构体变为 !Unpin，从而在编译期阻止安全代码获取可移动的可变引用。
- 移动 (move) 与地址敏感性：Rust 的所有权移动（如赋值、函数返回等）会改变值在内存中的位置，对于不依赖固定地址的类型（Unpin）无影响，但对自引用类型会破坏内部指针的有效性，因此需要通过 Pin 阻止移动。

### 📝 文章摘要

本文解释了 Rust 中 `std::pin::Pin` 的工作原理与必要性。`Pin` 是一种指针包装器，提供类型级保证：被指向的值不会通过该指针被移动，核心场景是保护自引用结构（如编译器生成的 `async` 状态机中的 `Future`）的内部引用不因移动而悬垂。文章指出，一旦自引用建立，移动该值会使内部原始指针指向旧地址，导致未定义行为；`Pin` 通过限制安全代码恢复 `&mut T` 的能力来防止移动，但对不依赖固定地址的 `Unpin` 类型无此限制。文章还介绍了构造 `Pin` 的几种方式，包括 `Pin::new`（仅限 Unpin）、`pin!` 宏（栈上固定）和 `Box::pin`（堆固定），并说明需通过 `PhantomPinned` 在自定义结构中显式声明 `!Unpin` 以维护安全契约。

## 7. [When Impressive Performance Gains Do Not Matter](https://blog.colinbreck.com/when-impressive-performance-gains-do-not-matter/)

来源：**Lobsters**

### 🔑 关键概念

- 注意力阈值：人因研究表明，保持用户注意力集中的延迟上限约为10秒。超过这个时间，用户会转而处理其他任务，使得进一步的响应时间优化对用户体验的改善不再显著。
- 从一到二的跨越：指受物理或逻辑约束的流程中，仅靠百分比的效率提升可能无法产生实际效益，必须突破某个临界点（如将每项任务耗时压缩到能够多完成一项完整工作的程度）才能带来根本性改变。
- 背压：在数据管道中，下游处理阶段速度变慢会向上游传递阻塞信号，防止系统过载。这意味着单个阶段速度的大幅提升，如果下游仍是瓶颈，则端到端的吞吐量不会改善。
