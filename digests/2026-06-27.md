## 2. [Fintech Engineering Handbook](https://w.pitula.me/fintech-engineering-handbook/)

来源：**Hacker News** · ⬆ **105** points · 💬 **32** comments

### 🔑 关键概念

- 幂等性键：用于确保网络重试不会导致重复操作（如重复扣款），客户端为每条请求生成唯一键，服务端据此识别并忽略已处理请求，是金融系统中防止金钱凭空产生的核心模式。
- 事件溯源：将所有对金钱状态的变更记录为不可变的事件序列，而非直接覆盖余额，从而提供完整的审计轨迹，并能随时重放事件重建任意时间点的状态。
- 最小单位精度：用整数存储货币的最小单位（如分、聪、Wei），避免浮点舍入误差，但需注意不同货币的精度位数不同（法币参考 ISO 4217，加密资产由代币自身定义），且大数值可能溢出 64 位整数。
- 对账：定期或不定期地将系统内数据与外部来源（银行、支付网关、区块链）交叉比对，识别并处理差异，是“不信任”原则的具体实现，防止任何单一数据源错误累积。
- 四舍五入残余跟踪：当金额拆分或计算后必须舍入时，各部分的加总可能不等于原总数，需将差额记入显式的“舍入账户”，避免凭空产生或丢失金钱。

### 📝 文章摘要

一篇名为《Fintech Engineering Handbook》的在线手册在 Hacker News 上被分享，它系统整理了以金钱为核心的软件工程中最重要的模式与原则。手册以三条原则贯穿始终：“不编造数据”（杜绝重复记录与任意余额更新）、“不丢失数据”（所有动账必须持久化且提供完整审计）、“不信任”（对外部、内部组件和世界均做验证）。内容涵盖货币的表示选择、精度处理、舍入策略与货币元数据控制等具体实践，并针对加密资产的特殊性提出了扩展方案。该手册定位为面向刚进入或已从事金融科技工程人员的参考文档与共享词汇表，目前作为活文档接受社区贡献。

### 💬 社区讨论

关于货币存储策略的争议较大：用户 xlii 强烈主张“始终使用整数”存储金额，认为其他方式会带来风险；lxgr 则警告不要在 API 数据格式中使用最小单位精度，因其在不同系统间可能产生解释歧义，但 antonymoose 以自己从事高频交易的经验反驳，称整数分在行业中被广泛使用且行之有效。部分评论认为手册内容并不局限于金融科技，benashford 指出幂等性、重试、事件顺序等措施

## 5. [NLNet Labs LLM Policy](https://nlnetlabs.nl/llm-policy/)

来源：**Lobsters**

### 🔑 关键概念

- LLM 生成内容限制：NLNet Labs 要求所有代码与文档贡献必须由人撰写，禁止直接提交 LLM 或概率工具生成的内容；唯一的例外是漏洞报告可以包含 LLM 建议的修复，以便项目方更快定位问题。
- 披露义务：在与项目互动（如创建 issue、PR、社区帖子或发送漏洞报告）时，若使用了 LLM，必须主动声明，让双方意识到潜在偏差，并尊重维护者的时间。
- 责任归属：即使将 LLM 用于代码检查、分析或评审，提交者仍完全对 LLM 输出的正确性负责，必须在分享前自行验证和理解信息。
- 翻译场景的警示：不鼓励用 LLM 进行日常交流翻译，因其生成特性容易导致误解；若使用机器翻译，建议声明以避免误传，或直接使用母语沟通。
- 漏洞报告的合规路径：政策允许提交经 LLM 发现的漏洞，但要求提交者先由人工核实问题与严重程度，再发送报告并声明使用了 LLM，否则可能被拒绝。

### 📝 文章摘要

NLNet Labs 于 2026 年 6 月 26 日修订并公布了其 LLM 使用政策，对项目贡献和社区互动中如何使用大型语言模型做出了严格限定。核心规定是代码与文档必须由人类创作，禁止直接提交 LLM 生成的内容，仅在漏洞报告环节允许附带 LLM 建议的修复来协助问题定位。政策同时要求所有沟通中若使用了 LLM 须主动披露，且使用者需对 LLM 输出的正确性负全责，尤其对翻译等场景持谨慎态度。这一举措反映出开源社区有意通过明确的规则，保障贡献的原创性、交流的真诚性，降低 AI 生成内容带来的质量与信任风险。
